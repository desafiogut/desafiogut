1. RESUMO EXECUTIVO
--------------------------------------------------------------------------------
O DESAFIOGUT precisa suportar 10k usuários simultâneos quando o contrato LeilaoGUT
for para a Mainnet (MC40). Esta análise leu os 42 itens da "LISTAGEM DE ESCALABILIDADE
COMPLETA" e confrontou CADA UM com o estado REAL do código (arquivo:linha), corrigindo
status incorretos e separando o que é código, configuração de plataforma, ou decisão
de arquitetura.

Compromisso: TODOS os 42 itens são tratados antes do MC40 — "tratar" significa
IMPLEMENTAR, CONFIGURAR ou DECIDIR-E-DOCUMENTAR (descope com justificativa). Um plano
honesto não implementa cegamente itens que não cabem na stack (ex.: Streaming SSR numa
SPA Vite): resolve-os com uma decisão arquitetural registrada, realocando o esforço
para os itens que de fato movem a agulha dos 10k usuários.

ACHADOS QUE MUDAM A LISTA (evidência):
  • Vários itens já estão PARCIAIS/QUASE-FEITOS — não "Pendente":
      - Item 2 (Vendor Chunking): vite.config.js:26-40 já isola `react` e `privy`+`@coinbase`.
      - Item 32 (WebSocket pooling): useRealtimeConfig.js:34-80 já remove canal no unmount,
        usa cliente singleton e backoff exponencial, e NÃO subscreve `table:'*'`.
      - Item 19 (Rate limit): 30/45 funções; e o limiter já é shared-store (Netlify Blobs,
        strong consistency) — arquiteturalmente correto p/ serverless (≠ in-memory).
      - Item 21 (N+1): `Promise.all` em 208 pontos; loops restantes são admin/cron/analytics.
      - Itens 35/36 (observability): Sentry (react+node) já integrado; falta só RUM/web-vitals
        e regras de alerta.
  • 3 itens NÃO se aplicam à stack atual (SPA Vite + Privy client-side, sem framework SSR):
      - Item 7 (Selective Hydration), Item 8 (Streaming SSR), Item 9 (Module Federation).
        → DESCOPE recomendado (ver §6). A skill react-performance confirma que dependem de
        RSC/SSR (Next.js), inexistente aqui. Migrar de framework antes do MC40 = risco enorme,
        ganho marginal para o gargalo real (bundle Privy + Realtime + DB).
  • Item 28: Read Replicas (Supabase) resolvem 10k leituras; "Multigres" (sharding tipo
    Vitess) é prematuro → implementar Read Replica, descope Multigres.

PRINCIPAIS GARGALOS REAIS (priorizados):
  1. Bundle Privy ~2.84 MB (859 KB gz) carregado no caminho crítico (itens 2,3,5,11,12).
  2. Conexões: Functions serverless sem pooler de transação → esgotam conexões Postgres
     em pico (item 27, CRÍTICO).
  3. Leituras quentes sem cache distribuído (config_remota, listagens) (itens 20,33,16,17).
  4. Realtime: 10k WebSockets simultâneos vs. limite do plano Supabase (itens 31,33-plataforma).
  5. Índices compostos/parciais ausentes em `lances` para apuração/consulta (itens 25,26).

  • vite-patterns (substitui a ausente @bundle-optimization) → Vite 8 usa `build.rolldownOptions`

    tabela de TTL (cache 5-15min, rate-limit = janela); Upstash REST (HTTP) evita exaustão de
    conexão TCP do par serverless+Redis; rate-limit sliding-window via Lua (EVAL).
  • postgres-patterns (substitui @database-performance/@index-advisor) → ordem de índice composto
    (equality→range valida (edicao_id, created_at)); covering INCLUDE; CONCURRENTLY zero-downtime;
    paginação por CURSOR (O(1) vs OFFSET); fila nativa via FOR UPDATE SKIP LOCKED (alternativa ao
    BullMQ); pg_stat_statements p/ monitorar queries lentas (item 36).

6. DECISÕES DE ARQUITETURA (RUFLO — Agente de Transação: acionável, sem regressão)
--------------------------------------------------------------------------------
DESCOPE com justificativa (resolução = decidir e documentar, não implementar):

  • Item 7 (Selective Hydration) e Item 8 (Streaming SSR):
    Dependem de SSR/RSC. O DESAFIOGUT é SPA Vite + Privy (auth client-side, exige browser).
    Implementar exigiria migrar para Next.js/Remix — semanas de retrabalho e ALTO risco de
    regressão (R1) às vésperas do MC40, com ganho marginal vs. resolver o bundle (itens 2/3/5).
    DECISÃO: não implementar pré-MC40. Reavaliar pós-Mainnet SE Web Vitals (item 35) provarem
    necessidade. O ganho de FCP/LCP vem do code-splitting (ONDA 2), não de SSR.

  • Item 9 (Module Federation / microfrontends):
    Padrão para múltiplos times/deploys independentes. DESAFIOGUT é um app, um time.
    Adiciona complexidade operacional sem ganho de escala para 10k usuários.
    DECISÃO: não implementar. Code-splitting por rota (item 3) entrega o benefício real (chunks
    sob demanda) sem o custo de microfrontends.

  • Item 28 — "Multigres": Read Replica (Supabase) resolve a carga de leitura dos 10k. Multigres
    (sharding horizontal tipo Vitess) é tecnologia nova e prematura para este volume.
    DECISÃO: implementar Read Replica + cliente `supabaseReadOnly` (escrita só no primário — R11);
    descope Multigres.

  • Item 14 (BullMQ) — DECISÃO DE HOST DO WORKER:
    Netlify Functions são efêmeras: não rodam um consumer BullMQ de longa duração.
    Opções (decidir antes da ONDA 7):
      A) Netlify Background Functions p/ a maioria das tarefas assíncronas (item 13) — sem Redis,
         mais simples; suficiente p/ notificações/consolidação. RECOMENDADO p/ o go-live.
      B) Fila NATIVA Postgres (postgres-patterns: FOR UPDATE SKIP LOCKED) consumida por uma Background
         Function/pg_cron — retry/DLQ sem Redis nem worker externo; reaproveita o Supabase já existente.
         RECOMENDADO se precisar de garantias de entrega sem subir infra nova.
      C) BullMQ + Redis com worker em host persistente (Fly.io/Railway/VM) — só se A/B não cobrirem
         prioridade/throughput; adiciona superfície operacional (evitar pré-Mainnet sem necessidade).
    Princípio: não introduzir worker externo sem necessidade comprovada.
