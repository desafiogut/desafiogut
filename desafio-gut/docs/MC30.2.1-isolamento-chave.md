# MC30.2.1 — Migração do Isolamento da Chave Mestra (Defender → Biconomy + KMS)

> PILAR 2 (EVERYTHING CLOUD CODE): documentação modular da arquitetura de
> assinatura isolada. Branch `feat/mc30.2.1`. Ver também `security_audit.md` §MC30.2.1
> e `MC30.1-isolamento-chave.md` (fundação).

## Motivação

O **OpenZeppelin Defender** entra em **sunset a 2026-07-01** (novos cadastros já
bloqueados). A infraestrutura de assinatura isolada do MC30.1 migra para a
**Biconomy** (`@biconomy/account`, ERC-4337), que resolve **nativamente** a
concorrência de nonces (crítico no leilão de menor lance único com lances
simultâneos), oferece **Paymaster** (subsídio de gás, UX de 1 clique) e evita o
custo de uma fila manual de transações. O contrato `Leilao.sol` **não é alterado**
(R10) — a autoridade da coordenação é transferida para o Smart Account via o
**two-step** já existente no contrato.

## Decisão-chave: owner via KMS (Q1 do plano · R12)

A Biconomy não tem HSM próprio como o Defender — precisa de um **owner signer** para
autorizar as UserOperations. Para preservar o isolamento do MC30.1, esse owner é uma
chave que vive num **KMS/HSM remoto** (AWS KMS recomendado; desenho agnóstico). A
chave privada bruta **NUNCA entra no processo** Node das Netlify Functions.

## Arquitetura

```
                         ┌─────────────────────────────┐
  contract.mjs ─┐        │      _lib/signer.mjs        │
  consolidar-   ├──────► │   (FACHADA DE ASSINATURA)   │
   lances.mjs   │        │  backendAssinatura()        │
  ia-preditiva  ┘        │  obterSignerCoordenacao()   │
                         └───────────┬─────────────────┘
                                     │ seleção por env (SIGNER_BACKEND)
              ┌──────────────┬───────┴───────────┬────────────────────┐
        local-key        defender            biconomy (ALVO MC30.2.1)
       (testnet)      (FALLBACK, sunset)   createSmartAccountClient
     ethers.Wallet   DefenderRelaySigner   ├─ owner: KmsSigner (KMS) ──► _lib/kms-signer.mjs
                                            ├─ Bundler ERC-4337         (AbstractSigner; chave no KMS)
                                            └─ Paymaster (opcional)     _lib/kms/aws-kms.mjs (AWS)
                                                     │
                                          BiconomySmartAccountSigner (ADAPTER)
                                          expõe o Smart Account como ethers v6 Signer
```

- **MC30.1:** `new Wallet(...)` só em `_lib/signer.mjs`; 3 call-sites via `obterSignerCoordenacao`.
- **MC30.2.1:** acrescenta o backend `biconomy`. Os **3 call-sites permanecem INALTERADOS**
  (achado #4, Opção A): o adapter expõe o Smart Account como um `ethers.Signer`, pelo que
  `new Contract(addr, abi, signer)` + `contract.metodo(...)` + `tx.wait()` continuam a funcionar.

## O adapter (achado #4)

`BiconomySmartAccountSigner extends AbstractSigner` (lazy):
- **Escritas** (`sendTransaction`): traduzidas em UserOperation via Bundler. O `tx`
  devolvido expõe o **hash REAL** da transação (resolvido por `waitForTxHash()`), para
  que `tx.wait()` (embrulhado pelo ethers no provider) e `provider.waitForTransaction(tx.hash)`
  (em `consolidar-lances.mjs`) funcionem contra a RPC real.
- **Leituras** (views: `edicaoNonce`, `coordenacao`): roteiam por `runner.provider`
  (`JsonRpcProvider` com `staticNetwork` derivado do chainId do bundler — evita um
  round-trip de deteção de rede por escrita).
- **`signTypedData`** (recibo EIP-712): delegado ao **owner KMS** → assinatura ECDSA
  recuperável à EOA owner, **idêntica** ao MC30.1 (achado #3).

## O owner KMS (`_lib/kms-signer.mjs` + `_lib/kms/aws-kms.mjs`)

`KmsSigner extends AbstractSigner`:
- `getAddress()` — deriva o endereço do DER SPKI (chave **pública**) do KMS.
- `signMessage(userOpHash)` — EIP-191; assina o digest crú (4337) via KMS.
- `signTypedData(...)` — recibo EIP-712, recuperável à EOA owner.
- O KMS devolve a assinatura em **DER**; o módulo normaliza **low-S** (EIP-2) e calcula
  o **recovery id `v`** para a assinatura `{r,s,v}` de 65 bytes do Ethereum.
- Provider agnóstico via `KMS_PROVIDER` (apenas `aws` implementado nesta fase).

## Mudança de endereço (achado #1)

Em ERC-4337 o `msg.sender` on-chain passa a ser o **Smart Account** (≠ EOA). Como
`Leilao.sol.apenasCoordenacao` exige `msg.sender == coordenacao` e é imutável, a
autoridade é transferida via two-step: `iniciarTransferenciaCoordenacao(<smartAccount>)`
(coordenação atual) + `aceitarTransferenciaCoordenacao()` (o próprio Smart Account, via
UserOperation). O `getCoordenacaoAddress()` reflete o novo endereço automaticamente
(cache preenchido na 1.ª assinatura).

## Seleção de backend

| `SIGNER_BACKEND` | `NETWORK_STAGE` | Backend ativo |
|------------------|-----------------|---------------|
| `biconomy`       | (qualquer)      | `biconomy`    |
| `defender`       | (qualquer)      | `defender`    |
| `local-key`      | (qualquer)      | `local-key`   |
| (vazio)          | `mainnet`       | `defender` (fallback, R11) |
| (vazio)          | sepolia/outro   | `local-key`   |

> O default mainnet permanece `defender` enquanto for mantido como fallback. Após a
> validação completa (SEGMENTO 8), o Defender é removido e o default passa a `biconomy`.

## Variáveis de ambiente

```
SIGNER_BACKEND=biconomy                 # ALVO de produção
RPC_URL=...                             # JSON-RPC (views + waitForTransaction)
BICONOMY_BUNDLER_URL=...                # Bundler ERC-4337 (obrigatório)
BICONOMY_PAYMASTER_URL=...              # Paymaster (opcional — subsídio de gás)
BICONOMY_API_KEY=...                    # opcional (paymaster sponsorship)
KMS_PROVIDER=aws|turnkey|fireblocks     # owner agnóstico (só 'aws' implementado)
KMS_KEY_ID=arn:aws:kms:...:key/...      # identificador da chave KMS (NUNCA a chave)
AWS_REGION=us-east-1
# COORDENACAO_PRIVATE_KEY: PROIBIDA em mainnet/biconomy — remover do env de produção.
```

Ver `frontend/.env.example` para descrições completas.

## Guarda de segurança (R9/R12 · ITEM 4.1)

`assertChaveBrutaAusenteEmMainnet()` — além da regra mainnet (MC30.1), com
`SIGNER_BACKEND=biconomy` recusa arrancar se: houver chave bruta em env; `KMS_PROVIDER`
for desconhecido; faltar `KMS_KEY_ID` ou `BICONOMY_BUNDLER_URL`.

## Cutover (ações humanas do operador — SEGMENTOS 7-9)

1. Provisionar a chave secp256k1 no **AWS KMS** (ECC_SECG_P256K1) + papel IAM escopado.
2. Configurar `BICONOMY_*` + `KMS_*` no Netlify; validar **handshake real** (KMS + Bundler).
3. Resolver o endereço do **Smart Account** e financiá-lo (ou configurar Paymaster).
4. **Transferir a coordenação** on-chain (two-step) e confirmar `coordenacao() == smartAccount`.
5. **Remover** `COORDENACAO_PRIVATE_KEY` e `DEFENDER_*` do env de mainnet do Netlify (R9).
6. SEGMENTO 8: remover o backend `defender` do código.
7. (Opcional, SEGMENTO 9) Migrar a coordenação para uma **Gnosis Safe** multisig.
