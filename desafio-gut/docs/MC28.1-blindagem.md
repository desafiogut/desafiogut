# MC28.1 — Blindagem de Privacidade dos Lances (Anti-Bot)

> Doc de arquitetura (PILAR 2 — EVERYTHING CLOUD CODE). Nota: o `cloud.md` do repo
> é específico dos fundos animados (MC26.1); esta funcionalidade é documentada aqui
> para não poluir um ficheiro de âmbito diferente.

## Objetivo

Esconder o valor real do lance enquanto o leilão está ativo, na **mainnet**,
sem degradar a UX (1 clique) nem regredir o legado Sepolia/localhost.
Abordagem **A2** — Híbrida com Compromisso Cego + Stateless Key-Per-Bid.

## Isolamento por ambiente (R9)

Toda a lógica está atrás de `NETWORK_STAGE === 'mainnet'` (backend) e
`import.meta.env.VITE_NETWORK_STAGE === 'mainnet'` (frontend). Em
Sepolia/localhost o comportamento é **byte-idêntico** ao legado.

## Fluxo de um lance (mainnet)

```
Utilizador ──(HTTPS, 1 clique)──► lance-relampago.mjs
   │  valida JWT + RBAC + saldo R$ (debitarSaldoRs)
   │  hash = keccak256(abi.encode(uint256 valor, address endereco))   ← espelha a EVM
   ├─► comprometerLance(idEdicao, endereco, hash)         [on-chain, coordenação assina]
   │       → só o HASH fica on-chain; valor invisível no mempool
   └─► gravarBid(bid:{edicaoId}:{endereco}:{rnd8})        [Blob Key-Per-Bid, imutável]
           → valor real off-chain, 1 chave por lance (sem race)

   resposta inclui commitmentHash → UI mostra "prova de participação"

FECHO (coordenação/admin) ──► consolidar-lances.mjs
   │  listarBids (paginação por cursor + leitura paralela em lotes)
   │  apura menor lance único OFF-CHAIN (Artigo VIII)
   │  assina EIP-712 (recibo) · lê edicaoNonce
   └─► consolidarResultado(idEdicao, vencedor, menorUnico)  via Flashbots Protect
           → O(1) on-chain · edicaoNonce++ (anti-replay) · marca blob consolidado
```

## Garantias de gás

- `comprometerLance`: O(1) (uma escrita de mapping + evento).
- `consolidarResultado`: O(1) (struct + `nonce++`), valor **singular** (sem arrays
  dinâmicos — ITEM 1.4).
- `darLance` legado: inalterado. `apurarVencedor` (view O(n)) mantido como
  fallback/auditoria, fora do caminho quente.

## Decisão `string` vs `bytes32` (ITEM 1.1)

`idEdicao` mantém-se `string`. O contrato Sepolia já deployado usa `string` e
R1/R9 exigem zero regressão; `bytes32` quebraria a ABI viva (9 ficheiros). bytes32
é otimização, R1 é absoluto → mantém-se `string`. Ver `security_audit.md`.
